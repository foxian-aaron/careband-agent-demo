# 智护环 CareBand Agent Demo 路演脚本

## 1. 开场介绍

大家好，我们展示的是“智护环 CareBand Agent”，一个面向长者照护场景的 AI Agent 状态观察系统。它不是普通健康 Dashboard，也不是聊天机器人，而是把长者状态、个人基线、照护事件和护工处理串成一个闭环。

## 2. 展示机构端

打开 `#/institution`。这里有 5 位长者的风险热力图，管理者可以看到高风险人数、需关注人数、稳定人数、待处理任务数和平均数据完整度。

## 3. 点击陈伯

点击陈伯进入 `#/elder/E001`。陈伯 78 岁，房间 203，有高血压标签，平时 7 日平均步数 2150 步，平均睡眠 6.5 小时。

## 4. 说明陈伯初始“需关注”

今天陈伯只有 820 步，睡眠 4.8 小时，晚药未确认。系统不是用通用阈值判断，而是和陈伯自己的个人基线比较，因此初始状态为“需关注”。

## 5. 触发“我有点头晕”

进入 `#/demo-control`，点击“触发陈伯头晕语音事件”。系统添加 20:15 的 voice_symptom 事件，原始文本是“我有点头晕”。

## 6. 展示状态升级为高风险

回到陈伯驾驶舱。Decision Trace 展示规则触发链路：活动明显下降、睡眠偏低、晚药未确认、老人主动反馈头晕，再叠加慢病标签，风险升级为“高风险”。

## 7. 展示护工待办

打开 `#/caregiver`。护工端出现高优先级任务“陈伯需要立即查看”，原因是“头晕反馈 + 晚药未确认 + 活动明显下降”，建议护工确认是否已进食和服药，并观察不适是否持续。

## 8. 展示家属安心卡

打开 `#/family/E001`。家属端不会展示复杂医学指标，而是温和说明：今晚 20:15 陈伯反馈有点头晕，系统已提醒护工，陈伯仍在长者中心内，未检测到跌倒。

## 9. 护工接单和完成

在 Demo 控制台或护工端依次点击“护工接单”、“确认晚药”、“完成护工处理”。系统写入备注：“20:25 护工A已查看陈伯，已确认晚药，陈伯目前在房间休息，建议明早继续关注活动和睡眠。”

## 10. 展示三端状态同步

回到家属端，显示“护工已查看，晚药已确认，正在持续观察”。回到机构端，陈伯任务状态从待处理变为“已跟进 / 持续观察”。

## 11. 总结产品价值

智护环的核心价值是把长者贴身状态感知、个人基线对比、多指标风险判断、三端照护展示和护工处理闭环连接起来。它不替代医生，也不做医疗诊断，而是帮助照护人员更早发现需要关注的人和事。
